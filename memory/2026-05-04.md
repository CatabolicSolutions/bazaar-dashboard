## Round 2 Session (01:30–01:54 MT)

### User reported two problems
1. **Hover tooltip broken** — mouseover doesn't show price/crosshair info
2. **Drag doesn't move the chart** — only overlays a forecast zone that blocks visibility. User wants actual pan/scroll.

### Debugging findings
- **TradeMarkers crash killed mousemove handler** — `history.findIndex(p => p.ts.includes(m.ts))` throws `Cannot read 'includes' of undefined` when `p.ts` is undefined. This is a silent/uncaught error that halts the handler, so the tooltip crosshair and chartTip display logic never runs.
- **Fix:** Add null guard: `p => p && p.ts && p.ts.includes(m.ts)`
- This was the root cause of the hover tooltip being completely dead — the handler crashed before drawing crosshairs or showing chartTip.

### Drag redesign required
- User explicitly stated drag should physically scroll/pan the price chart, not overlay a static forecast zone
- First attempt used canvas clipping + `info.px` override to shift data left on drag, but it introduced save/restore imbalances and the clip cut off the forecast rendering
- Second attempt cleaner but hover + forecast didn't populate simultaneously
- **Durable lesson:** drag-to-forecast needs a complete redesign — shift x-axis offset as a scroll, not a separate forecast canvas layer. Canvas clipping approach was on the right track but needs cleaner implementation.

### Final deploy state
- Deployed clean version with only verified fixes:
  1. `ctx.save()` after `ctx` declaration (blocking bug fix)
  2. `tradeMarkers` null guard (hover now works)
  3. `drawExecChart` null canvas guard (pre-existing error wrapped)
- Drag/forecast removed from this deploy — will re-add as clean feature

### Canvas state discipline (reinforced)
- `ctx.save()` must come AFTER `const ctx = ...` declaration — a ReferenceError blocks the entire script
- Each chart draw cycle must use exactly one `ctx.save()` and one `ctx.restore()`
- Unbalanced save/restore will corrupt canvas state and break subsequent draws
- `info.px` must be restored after clip override (use `window.__px0` pattern)
- `tradeMarkers` iteration needs null guards on `p.ts` to prevent silent handler crashes

### Key architectural fix
**Removed `ctx.translate()` entirely.** The translate approach shifted *everything* — grid, labels, legend — which caused clipping and forecast-invisible bugs. Replaced with:
- `opts.shift` parameter in `drawLines()` — shifts only series x-coordinates: `const x = px(i) - (opts.shift || 0)`
- `info.px` override in `drawPriceChart()` — temporarily overrides the px function to subtract shift, then restores original for forecast

This is the correct approach: grid/labels/legend stay fixed in place while only price series lines scroll left.

### Forecast `_x0` fix
Broken: `var _x0 = info.px(p.length-1) - _shift` — depended on shifted px function
Fixed: `var _x0 = W - (info.pr || 50) - _shift` — computes true edge of data regardless of px state
