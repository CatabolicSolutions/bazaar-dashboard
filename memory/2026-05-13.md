## DirectDEX Multi-Hop Fix (the actual fix)

### Context
Previous session claimed DirectDEX was fully fixed for all directions, but only CBBTC‚ÜíWETH single-hop was verified. WETH‚ÜíCBBTC multi-hop (via USDC) silently used the broken `exactInput` encoding and reverted on first live execution (nonce 371).

### What Actually Happened
- 00:54 MT: Cross-rate reversal triggered WETH‚ÜíCBBTC rotation
- DirectDEX nonce 371 reverted (unknown reason ‚Üí Bloc notification)
- 1inch fallback (nonce 372) succeeded ‚Üí now holding ~0.0023 CBBTC
- Bot properly fell back to 1inch when DirectDEX failed ‚Äî architecture works as designed for failures

### Root Cause
The Base SwapRouter02 (pre-Permit2 codebase) does NOT support multi-hop `exactInput` with any encoding variant tested:
- 4-param `exactInput((bytes,address,uint256,uint256))` ‚Üí InvalidFEOpcode
- 5-param `exactInput((bytes,address,uint256,uint256,uint256))` with deadline ‚Üí revert
- Same with offset ‚Üí revert

### The Fix
Replaced multi-hop `exactInput` with **two sequential single-hop `exactInputSingle` calls**:
1. Submit leg 1 (WETH‚ÜíUSDC) signed internally via private_key ‚Üí wait for confirmation
2. Return leg 2 (USDC‚ÜíCBBTC) as normal quote ‚Üí submitted by legacy_bridge flow

Changes made:
- `direct_dex.py`: New `_quote_and_build_multi` that signs/submits leg 1 internally, returns leg 2
- `direct_dex.py`: Removed dead `_build_multi_tx` method
- `env.py`: `dex_direct_config()` now passes `private_key` to `DexTransportConfig`

### Git Status
- Committed as `6b9e343` ‚Äî "fix multi-hop: sequential single-hop exactInputSingle instead of broken exactInput"
- Bot restarted and running live, holding CBBTC at ~0.0023

### Durable Lessons

## Real Root Cause of All DirectDEX Reverts (discovered 07:58-08:10 MT)

### Discovery
Debugging nonces 373, 375, 377 calldata revealed the **actual** root cause:
- Calldata decoded to `amount_in = 400099609399621000000000000000000` ‚Äî that's ~400 trillion WETH (~8 √ó 10^36 wei)
- Wallet had only 0.0004 WETH

### Root Cause: Double-Decimal Conversion
The bug was in `direct_dex.py` ‚Üí `get_swap_quote()`:
- Line 363: `amount_int = _amount_to_int(from_addr, amount)`
- `amount` was ALREADY in smallest unit (wei/sats) from the preflight
- `_amount_to_int` converted it AGAIN (multiplied by 10^18)
- Result: every DirectDEX swap tried to spend ~10^18 √ó the actual balance

### Impact
ALL 4 DirectDEX attempts failed for this single reason, not separate bugs:
- Nonce 371: WETH‚ÜíCBBTC (original code) ‚Üí amount = 0.0804 √ó 10^18 √ó 10^18
- Nonce 373: leg1 WETH‚ÜíUSDC (new multi-hop code) ‚Üí same bug inherited
- Nonce 375: leg1 CBBTC‚ÜíUSDC ‚Üí same bug
- Nonce 377: leg1 WETH‚ÜíUSDC ‚Üí same bug

### The Fix
Changed line 363 to `amount_int = amount  # already in smallest unit`
Committed and deployed. Next DirectDEX rotation should use correct amounts.
