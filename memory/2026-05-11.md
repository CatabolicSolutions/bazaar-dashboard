# 2026-05-11 — Continuous Rotation Model Implementation

## Session Summary
Implemented Conor's approved rotation cycle design: bot evaluates all three positions (WETH, BTC, USDC) every tick with continuous entry_scores, routes directly between assets without USDC pass-through when the target has adequate shaping.

## Key Durable Decisions

### New Rotation Architecture (committed)
- **Continuous entry_score** — `analyze_asset()` now always computes entry_score (not gated by `buy_ready`). Score uses band_depth (45%), edge_q (30%), conv_q (20%), trend_b (5%).
- **Direct rotation in WETH/BTC sides**: sell/stop signals check the other asset's score. If >= FORCED_EXIT_MIN (0.15), rotates directly (ROTATE_TO_BTC/ROTATE_TO_WETH). If both are bad, exits to USDC.
- **Proactive rotation**: if the other asset's score exceeds current asset's score by >= PROACTIVE_EDGE (0.15) and the other is >= ASSET_VIABLE_MIN (0.25), rotates preemptively.
- **USDC entry gating**: uses score >= ASSET_VIABLE_MIN (0.25) instead of buy_ready. If both viable, enters highest score.
- **Action types added**: `ROTATE_TO_BTC` (target=Side.CBBTC), `ROTATE_TO_WETH` (target=Side.WETH) — mapped in `_legacy_action_to_side()`.

### Files Modified
- `bot/dual_rotator.py` — `analyze_asset()`: entry_score computed unconditionally; `decide()`: replaced whole function body with continuous rotation model
- `runtime/state/reconcile.py` — `WETH_DUST` set to 0.0001 (fixed dust oscillation)
- `runtime/strategy/live_bridge.py` — `_legacy_action_to_side()`: added ROTATE_TO_BTC→CBBTC, ROTATE_TO_WETH→WETH mappings

### Remaining (uncommitted)
- `dual_rotator.py` still has a hanging duplicate return line in the USDC HOLD block (line ~434). Fixed locally but not committed.

### Previous Work (still relevant)
- WETH_DUST alignment fix (0.0001) prevents tick-by-tick side flips that reset consolidation_bars
- Consolidation bars persist override after `_persist_back()`
- Binance primary price feed (CoinCap fallback, CoinGecko tertiary)
- Backtest proved 21 entries / 21 exits at 5-min resolution across May 7-10 (net +0.01%, 505/721 bars USDC)

### Design Constraints
- No deployment to droplet — only implementation and local testing
- User to test with Athena/Vigil/Harness/Matrix quarterly
- Git-only edits (no sed patches)
- Token efficiency: minimal noise in replies

## Later Session — Bot Goes Live (First On-Chain Swap)

### Boot Loop & Env Var Fix
- **Systemd rejected `1INCH_API_KEY`** — Environment variable names starting with digits are invalid in systemd. The boot logged `Environment has a bad assignment: 1INCH`. Fixed by renaming to `INCH_API_KEY` in both `.env` on droplet and `runtime/integrations/env.py`.
- **Commit**: `b0912ab` — "fix: rename 1INCH_API_KEY to INCH_API_KEY (systemd rejects digit-starting env vars)"
- **Deployed**: `git pull origin master && systemctl restart uniswap-rotator.service` on droplet. Boots cleanly now.

### Execution Amount Bug (USDC Swap Failed)
- **Root cause**: `runtime_loop.py` used `int(before.portfolio_usd)` as swap amount. But `portfolio_usd` includes total USD value (including WETH/CBBTC dust), exceeding actual USDC wallet balance. 1inch API returned: `"Not enough balance. Amount: 228000000. Balance: 227503054"`.
- **Fix**: Added `_derive_execution_amount()` method returning `int(snapshot.usdc)` when side_before is USDC, `int(portfolio_usd)` otherwise.
- **Commit**: `7ee4053` — "fix: use wallet USDC balance instead of portfolio_usd when swapping from USDC"

### First Live Swap — USDC → WETH ✅
- **Timeline (2026-05-11 09:38 UTC)**: Allowance check passed → 1inch quote received → Swap submitted → Transaction sent on-chain → Dashboard snapshot confirmed `side: WETH`, `weth: 0.0965`
- **Result**: Bot successfully swapped ~$227 USDC to ~0.0965 WETH. First ever completed live swap.
- **Post-swap**: Bot now holding WETH, evaluating every 3s tick with corrected strategy (entry_score using band_depth/edge_q/conv_q/trend_b).

### Stdout Buffering Fix
- **Problem**: Live ticks were buffered in memory for minutes before flushing to journald (slow terminal on remote host).
- **Fix**: Added `os.environ["PYTHONUNBUFFERED"] = "1"` at top of `live_runner.py` + `sys.stdout.reconfigure(line_buffering=True)` fallback.
- **Commit**: `beba80a` — "fix: force unbuffered stdout for live_runner.py"

## Late Session — Exit Target Bug & Consolidation Gate Fix (2026-05-11 ~20:00 MT)

### Critical Bug: Exit / Stop Targets Anchored to EMA (not Entry Price)
**Problem**: `exit_p` and `stop_p` were calculated relative to EMA12/EMA50:
- `exit_p = e12 * (1 + trigger / 100.0)` — exits at trigger % above EMA12
- `stop_p = e50 * (1 - STOP_LOSS_PCT / 100.0)` — stops at stop-loss % below EMA50

When price rises, EMA12 chases it up. So exit target also rises — perpetually ~0.15% above current price. Bot never reaches the target because the target moves with price. Same for stop loss: as price pulls back, EMA50 drops lower, making the stop loss even further below current price. Both targets float away indefinitely.

**Fix deployed to droplet**: Changed both to use entry price as anchor:
- `exit_p = entry_price * (1 + trigger / 100.0)` — exit at trigger % above entry
- `stop_p = entry_price * (1 - STOP_LOSS_PCT / 100.0)` — stop at stop-loss % below entry
- `entry_price = state.get(f'entry_{key}', p)` — captured from state at position entry

**Impact**: The bot now has a fixed target for each position. When price returns to entry + 0.15%, the bot exits. It doesn't chase. This is the correct behavior Conor described: "We care about the return made from ENTRY PRICE to EXIT PRICE. Then we RELOAD and do it over. AGAIN AND AGAIN."

**Conor's note**: This is exactly the kind of bug that daily Vigil/Athena refinement reviews would catch in 5 minutes. A simple reading of the exit logic makes the problem obvious — exit should anchor to entry, not a moving average.

### Consolidation Gate Fix
**Problem**: `consolidation_bars` was set to `0` when side == USDC (or computed from state), meaning full aggressiveness ramps up over 60 ticks (~60 min for old 1-min loop). Combined with the exit target bug, the bot was effectively frozen.

**Fix deployed to droplet**:
- Changed line 325: `consolidation_bars = CONSOLIDATION_FULL_BARS  # always max — act every tick`
- This is a one-line change in `bot/dual_rotator.py`

**Impact**: Every tick evaluates with fully-open triggers (highest entry tolerance, closest exit targets). Bot no longer waits 60 minutes for aggressiveness to ramp up.

### Swap Execution Amount Bug (identified, not fixed)
**Problem**: `runtime_loop.py` calls `int(before.portfolio_usd)` as `amount_in` for swaps. When the wallet has $225 total (0.097 WETH + $1.50 USDC) and the bot tries to sell WETH, it passes 225 as raw units. The contract interprets this as 225 WETH when the wallet only has 0.097 WETH.

**Root cause**: The adapter assumes `amount_in` is always in token units (wei for ERC-20), but the caller passes portfolio_usd (a dollar value). For USDC→token swaps, `int(portfolio_usd)` happens to work because USDC is 1:1 with USD. But for token→USDC swaps, `int(portfolio_usd)` is way too large.

**Fix needed**: `amount_in` should be converted:
- When selling WETH: `amount_in = int(weth_balance_in_wei)` (not portfolio_usd)
- When selling CBBTC: `amount_in = int(cbbtc_balance_in_wei)` (not portfolio_usd)
- When selling USDC: `amount_in = int(usdc_balance)` (portfolio_usd is close enough when only holding USDC)
