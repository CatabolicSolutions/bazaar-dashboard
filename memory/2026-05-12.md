- Full parameter sweep backtest run over 721 five-minute bars (May 7-10)

- Wallet: `0xFf9cc57be71A7b851A422a851600ac23B3843e27`
- Server: `root@137.184.144.196`, bot root `/var/www/uniswap_rotator`
- Position size: ~$188 (0.0813 WETH at ~$2311/ETH)

### Durable Truths

#### Real Per-Swap Cost (from on-chain data)
- Each 1inch swap leg: ~0.35% slippage (aggregator routing fee)
- Round trip (WETH→CBBTC→WETH): **0.704%** of position size
- Gas on Base: negligible (~$0.001/tx at 0.01 gwei)
- This is the structural cost floor for any trade

#### Backtest Results (Parameter Sweep)
- **Entry scoring has genuine edge**: Gross PnL positive at every VOL_FLOOR setting tested (0.15% to 3.0%)
- **Cost kills everything below VOL_FLOOR=1.0%**: Even 100% win rate loses money at VOL=0.15% because avg gain (0.25%) < round-trip cost (0.704%)
- **Only VOL_FLOOR=1.0%/STOP_LOSS=1.0% breaks even**: +0.53% net over 4 days, 5 trades, avg gain 0.81% > 0.704% cost
- **Win rate drops with higher VOL_FLOOR**: 100% at 0.15% → 60-86% at 1.0-1.5% → but net PnL flips positive because winners are large enough
- **Gross PnL per trade scales roughly linearly with VOL_FLOOR**

#### Structural Problem: Post-Exit Comparative Scoring
- After WETH hits exit target and the position closes, EMA12 is now trailing behind the peak price
- Score = anchor_to_EMA12 / vol_floor → artificially low right after a winner
- BTC, having done nothing, looks "better" by comparison even though it has no real signal
- Bot chases the comparison → rotates into a suboptimal position because the just-profitable asset appears bad
- **Fix needed**: post-exit cooldown in USDC before re-scoring both assets

### Key Decisions Made
1. **Bot will remain stopped** until thresholds deployed with mathematical confidence
2. **VOL_FLOOR=1.0%, STOP_LOSS=1.0%, FORCED_EXIT_MIN=0.15%** identified as minimum viable configuration
3. **Post-exit cooldown** structure required — bot must rest in USDC after taking profit, not immediately compare both assets
4. **Cost reduction (Path B)** — investigate direct DEX routing vs 1inch aggregator to get round trip below 0.35%

### User Insight (Conor, 23:49-23:55 MT)

- After WETH TP'd and bot cycled into BTC, that move wasn't "reconciling into a better market" — it was chasing the comparison
- Conor's assessment is correct; the design flaw is structural, not a tuning issue

### Note on Confirmation Bars
- SELL_CONFIRM_BARS=3 at 3s resolution = 9 seconds of price confirmation
- Insufficient to filter tick-level noise for a 5-minute timeframe edge
- Either increase bars to 20-60 (1-3 min) or accept that 5-min bar backtests show the real signal dynamic while tick-level execution adds noise

# 2026-05-12 02:26-03:20 MDT — Force Trade via Cross-Rate + Fallback Execution

## Session Outcome
**Trade executed**: CBBTC→WETH rotation via DirectDEX → 1inch fallback on Base.
Wallet now holds 0.0815 WETH (~$186). Bot running on DEX_MODE=1inch.

## Bugs Found and Fixed

### Bug 1: cross_peak/cross_trough not persisted between bot cycles
`_build_legacy_state()` did not copy `cross_peak` or `cross_trough` from `RotatorState.extra` into the legacy dict.
`_persist_back()` did not save them back.
**Effect**: cross-rate peak-trailing was completely broken — peak reset to `entry_cross` every tick, reversal never detected.
**Fix**: Added both fields to `_build_legacy_state()` and `_persist_back()` in `runtime/strategy/live_bridge.py`.

### Bug 2: cross_peak/cross_trough reset on rotation fire prevents retry
`dual_rotator.py` line 525: `state['cross_peak'] = current_cross` resets the peak immediately when rotation fires, BEFORE the swap confirms.
If swap fails, next cycle has peak=current → no reversal detected → no retry.
**Fix**: Temporary NOP on line 525 (commented-out reset). Peak persists until swap confirms — bot retries every cooldown period.

### Bug 3: _derive_execution_amount() returned wrong CBBTC/WETH amount
`max(1, int(snapshot.cbbtc * 0.995))` → for 0.0023 CBBTC: `int(0.0023) = 0` → `max(1, 0) = 1` → `to_raw_amount(1, CBBTC[8dec]) = 100,000,000 raw` → 1inch rejected: "Not enough CBBTC balance (need 100M, have 232K)"
**Fix**: `return float(snapshot.cbbtc * 0.995)` preserves decimals. `to_raw_amount(0.0023, CBBTC) = 230,000 raw` → balance sufficient.

### Bug 4: fallback_transport not stored on self
`fallback_transport = OneInchTransport(...)` assigned to LOCAL variable, but `runtime_loop.py` checks `getattr(self, "fallback_transport", None)`.
**Effect**: Fallback never fired — both lookups returned None.
**Fix**: Changed to `self.fallback_transport = OneInchTransport(...)`.

### Bug 5: ZeroDivisionError when cross_trough=0
After fresh rotation to WETH, `cross_trough`=0 in state. HOLDING WETH code computes `reversal_pct = (current - trough) / trough * 100` → ÷0 crash.
**Fix**: Added guard `if cross_trough > 0 else 0.0`.
