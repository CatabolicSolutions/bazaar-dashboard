## Root Causes of Zero Trades Since May 7

**Blocker 1: Stale side label**
- Runtime state says side=WETH (from old dry-run simulation on May 7)
- Wallet has $227.50 USDC with only dust WETH/CBBTC
- `decide()` only checks EXIT conditions for WETH side branch. ENTRY logic (USDC branch) is never evaluated.
- Result: bot is stuck in a "hold WETH" loop forever, never looking for entries.

**Blocker 2: No consolidation timer (never built)**
- `conv_ok` requires `anchor <= CONVERGENCE_ENTRY_TOLERANCE_PCT` (0.08% WETH / 0.06% BTC)
- In an uptrend, price is 25-30$ above EMA12 (~1-1.2%), so anchor is always >> 0.08%
- `discount_ok` requires `p <= entry_t` (price below EMA12 - trigger%) — impossible in uptrend
- So `buy_ready` is NEVER True for either asset in a trending market
- The "consolidation timer" concept: when in USDC and price stays above EMA for many bars, the CONVERGENCE_ENTRY_TOLERANCE should widen so the bot can eventually enter on trend pullbacks.
- This mechanism was discussed multiple times but never coded into dual_rotator_v2.py. Need to build it foundationally.

**Blocker 3: Entry thresholds designed for range-bound, not trending**
- VOL_FLOOR/CAP and CONVERGENCE thresholds were tuned for a sideways market
- The rotate signal detection also requires 0.06%+ momentum differential AND 0.03%+ spread deviation — neither fires in a slow drift

3. **Convergence tolerance widens** from 0.08% toward 0.8% over full consolidation
4. **Entry trigger shrinks** by 30% at full consolidation (entry target moves closer to current price)
5. Uses `CONSOLIDATION_FULL_BARS` (60 ticks ≈ 3 min) as the ramp denominator, not BOOTSTRAP_BARS (10)
6. Removed `price > EMA` condition from consolidation increment

## Session 2026-05-10 22:10-23:03 MT: WETH Dust Bug — Real Root Cause of Stuck Consolidation

### Root cause of consolidation_bars stuck at 1
After hours of diagnostic patches (most of which had SYNTAX ERRORS due to nested double-quotes inside f-strings, causing crash loops), the real bug was found:

**`WETH_DUST = 0.000001` (1 micro-WETH) in `runtime/state/reconcile.py`** did not match the `_reconcile_side()` check threshold of `0.0001` (100 micro-WETH) in `live_bridge.py`.

The bot's wallet had trace WETH dust (~0.00005 WETH) from a prior swap — between the two thresholds. Every single tick:
1. `run_reconciliation_cycle()` calls `infer_side_from_balances()` → sees 0.00005 WETH > 0.000001 → returns `Side.WETH`
2. This doesn't match the state file's `Side.USDC` → reconciliation resets `bars_since_flip = 0` and saves state as `Side.WETH`
3. `decide_target_side()` loads state (now `side='WETH'`), calls `_reconcile_side()` → sees 0.00005 WETH < 0.0001 → returns `Side.USDC`
4. Side changes → resets `bars_since_flip = 0` AND `consolidation_bars = 0`
5. `decide()` returns `consolidation_bars = 1` (0 + 1)
6. Save → file has `consol_bars = 1`
7. Repeat forever → tick advances, consol_bars never exceeds 1

**Fix (commit `bcd1d9a`):** Changed `WETH_DUST = 0.0001` in `reconcile.py` to match `_reconcile_side()` threshold. Also tightened `CBBTC_DUST = 0.0001` for symmetry.

### Why all diagnostic patches failed
Nearly every f-string diagnostic had `_ex.get("consolidation_bars","?")` — nested double-quotes inside a double-quoted f-string. Python parser sees the `"` before `consolidation_bars` as ending the format string. SyntaxError → module fails to compile → service crash-loop with restart counter increments.

The only way to write this safely:
- Use single quotes for the OUTER string: `f'cb={d.get("consolidation_bars","?")}'`
- Or use `chr()` concatenation for keys
- Or write diagnostics to a FILE instead of stderr/stdout to avoid stream capture issues

### Durable lessons from this session
- **Stop injecting diagnostics via sed/heredoc** — nested quotes in f-strings cause SyntaxError → crash loop. Use single-quoted f-strings or file-based diagnostics.
- **Always check the SIMPLE explanation first** — the dust threshold mismatch was a 1-line fix that took 6 hours of chasing because I assumed the persistence layer was wrong instead of checking reconciliation thresholds.
- **When a value resets every tick but saves correctly, look for reconciliation code** that runs before the decision code, not in the persistence layer.
- **Side reconciliation + dust mismatches can create infinite loops** — one threshold says "you're in position X", another says "you're not", flip every tick.
- `bars_since_flip` and `consolidation_bars` both reset to 0 in `decide_target_side()` when `_reconcile_side()` returns a different side — this is intentional and correct behavior, but the thresholds feeding reconciliation MUST be aligned.
